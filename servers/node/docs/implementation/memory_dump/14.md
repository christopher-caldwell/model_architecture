# Memory Dump 14: Restore Rust Field Names Across Node Layers

## Task
The user noticed GraphQL fields had been renamed to JavaScript-style `createdAt`/`updatedAt` conventions. The correction was to preserve the Rust data field names across domain and transport contracts.

## Changes Made
- Renamed domain entity and payload data fields to Rust names:
  - `dt_created`, `dt_modified`
  - `author_name`
  - `book_id`, `book_copy_id`
  - `full_name`, `max_active_loans`
  - `member_id`, `member_ident`
  - `dt_due`, `dt_returned`
- Renamed application command input fields:
  - `member_ident`
  - `book_copy_barcode`
- Updated application command implementations and tests to use the renamed fields.
- Updated persistence mappers and write-repository return objects to produce Rust-shaped domain entities.
- Updated HTTP request/response DTO internals and route command calls to use Rust field names.
- Updated GraphQL DTOs, Pothos object fields, input fields, and resolver argument mapping to expose Rust field names.
- Removed stale generated `.js`/`.d.ts`/`.map` artifacts from `packages/domain/src`.
- Cleaned stale `dist` output and rebuilt so compiled artifacts no longer contain old camelCase field names.

## Architectural Notes
- Function and class names remain idiomatic TypeScript where they are not serialized data fields.
- The field-name preservation applies to data shapes: domain entities, creation/prepared payloads, application command inputs, persistence rows/mappers, HTTP bodies, and GraphQL objects/inputs/args.
- GraphQL operation names remain as previously implemented. The corrected GraphQL object/input fields now match the Rust struct field names.

## Verification
- `pnpm typecheck` passed after a clean rebuild.
- `pnpm test` passed: 3 test files, 8 tests.
- `pnpm lint:boundaries` passed.
- Grep confirmed the old camelCase data-field names are absent from `packages` outside ignored TypeScript build metadata.
- Printed runtime GraphQL schema confirmed fields including:
  - `CatalogTitle`: `author_name`, `dt_created`, `dt_modified`, `isbn`, `title`
  - `LibraryMember`: `dt_created`, `dt_modified`, `full_name`, `max_active_loans`, `member_number`, `status`
  - `LoanRecord`: `dt_created`, `dt_due`, `dt_modified`, `dt_returned`, `loan_number`

## Next Context
Use Rust field names for all data structures going forward. If adding fields from the Rust code, copy the field names exactly into domain/application transport DTOs unless the user explicitly asks for idiomatic JS naming at that boundary.
