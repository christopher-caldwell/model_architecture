## Domain Design

The TypeScript domain avoids Rust-specific newtype ceremony. IDs are type aliases (`number` or `string`) and entities are interfaces.

Domain behavior is represented as functions that accept entities and either return the next state or throw a domain error:

- book copy guards/transitions:
  - `ensureBookCopyCanBeBorrowed`
  - `sendBookCopyToMaintenance`
  - `completeBookCopyMaintenance`
  - `markBookCopyLost`
  - `markBookCopyFound`
- member guards/transitions:
  - `suspendMember`
  - `reactivateMember`
  - `ensureMemberCanBorrow`
  - `ensureMemberWithinLoanLimit`
- loan guard:
  - `ensureLoanCanBeReturned`

Creation preparation functions preserve the Rust intent without copying its shape exactly:

- `prepareBook`
- `prepareBookCopy`
- `prepareMember`
- `prepareLoan`

## Ports

Domain exports repository interfaces:

- `BookWriteRepository`, `BookReadRepository`
- `BookCopyWriteRepository`, `BookCopyReadRepository`
- `MemberWriteRepository`, `MemberReadRepository`
- `LoanWriteRepository`, `LoanReadRepository`

Domain also exports:

- `WriteUnitOfWork`
- `WriteUnitOfWorkFactory`

The write unit of work exposes transaction-scoped write repositories plus `commit()` and `rollback()`.

## Architectural Notes

This keeps the core framework-free. No package imports from `@library/*` exist inside domain. Business rules exist in the domain as reusable guards/transitions, but multi-aggregate workflows still belong in application commands.

Transport layers must not call these domain transition functions directly. They should call application commands/queries only.
