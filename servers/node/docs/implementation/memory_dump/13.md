# Memory Dump 13: Split GraphQL Operations Into Bounded Subsections

## Task
Refactor the Node Pothos GraphQL schema so operations are grouped like the Rust GraphQL layer:

- `catalog/types.ts`, `catalog/queries.ts`, `catalog/mutations.ts`
- `lending/types.ts`, `lending/queries.ts`, `lending/mutations.ts`
- `membership/types.ts`, `membership/queries.ts`, `membership/mutations.ts`

The intent is organizational parity with the Rust architecture, not a 1:1 translation.

## Changes Made
- Replaced the monolithic `packages/graphql-server/src/schema.ts` with a small schema assembly module.
- Added `packages/graphql-server/src/schema-builder.ts` for shared Pothos builder, GraphQL context, and `safeResolver`.
- Moved Pothos output/input definitions into bounded-area `types.ts` files.
- Moved query registrations into bounded-area `queries.ts` files.
- Moved mutation registrations into bounded-area `mutations.ts` files.
- Kept shared GraphQL DTO mapping in `packages/graphql-server/src/types.ts`.
- Kept resolvers thin: they parse GraphQL args, call application commands/queries through `ctx.deps`, and map results to GraphQL transport types.

## Architectural Decisions
- Pothos field registration uses side-effect imports from `schema.ts`, which keeps each bounded area self-contained while preserving one shared schema builder.
- Cross-area GraphQL type reuse is transport-level only. Example: lending mutations can return `InventoryCopyRef` for `reportLostLoanedBookCopy`, matching the Rust GraphQL behavior without importing application/domain logic.
- Error mapping remains centralized through `safeResolver` and `mapGraphqlError`.
- No domain or application packages were imported by GraphQL subsection files. The GraphQL transport still imports only `@library/server-bootstrap` and `@library/auth-core` at the package boundary.
- Input ref exports use explicit portable inferred aliases to avoid TypeScript declaration errors from Pothos internals.
