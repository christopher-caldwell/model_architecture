## Architectural Decisions

The package graph intentionally mirrors the Rust crate graph:

- `domain`: imports no workspace packages.
- `application`: imports `domain`.
- `auth-core`: imports no workspace packages.
- `persistence`: imports `domain` and `application`.
- `server-bootstrap`: imports `application`, `auth-core`, and `persistence`.
- `http-server`: imports `server-bootstrap` and `auth-core`.
- `graphql-server`: imports `server-bootstrap` and `auth-core`.

TypeScript project references were added to make package boundaries explicit at build time. A custom `scripts/check-boundaries.mjs` was added because the architecture rule is important enough to enforce independently of TypeScript compile success.

The scaffold uses `NodeNext`, strict TypeScript settings, declaration output, and ESM packages. Placeholder `index.ts` files are empty exports so the packages are syntactically valid before implementation.

## Next Task Context

Task 3 is domain implementation. Build the domain as an idiomatic TypeScript model, not a Rust newtype clone.

Rust domain facts to preserve:

- Book: id, ISBN, timestamps, title, author name. Creation payload prepares write-ready book data.
- Book copy statuses: `active`, `maintenance`, `lost`.
- Book copy transitions:
  - active can be borrowed
  - active -> maintenance
  - maintenance -> active
  - non-lost -> lost
  - lost -> active
- Member statuses: `active`, `suspended`.
- Member transitions and guards:
  - active -> suspended
  - suspended -> active
  - active members can borrow
  - active loan count must be below max active loans
- Loan:
  - active loan has no returned timestamp
  - already returned loan cannot be returned again
- Domain ports include read/write repository interfaces and write unit-of-work interfaces.
