## Persistence Design

Read repositories take a `pg.Pool`:

- `BookReadRepositoryPostgres`
- `BookCopyReadRepositoryPostgres`
- `MemberReadRepositoryPostgres`
- `LoanReadRepositoryPostgres`

Write repositories take a transaction-scoped `pg.PoolClient`:

- `BookWriteRepositoryPostgres`
- `BookCopyWriteRepositoryPostgres`
- `MemberWriteRepositoryPostgres`
- `LoanWriteRepositoryPostgres`

`SqlWriteUnitOfWorkFactory`:

- acquires a client from the read-write pool
- starts `BEGIN`
- returns `SqlWriteUnitOfWork`

`SqlWriteUnitOfWork`:

- exposes transaction-scoped write repos
- commits once with `COMMIT`
- rolls back once with `ROLLBACK`
- releases the client after finishing

## SQL Notes

The SQL is inline in `src/sql.ts`, grouped by aggregate and command/query role. It preserves Rust behavior:

- read queries do not lock
- write reads use `FOR UPDATE`
- active loan sentinel remains `'9999-01-01 00:00:00+00'::TIMESTAMPTZ`
- loan identifiers are generated as `LN-000001` style values
- status lookup still uses `library.struct_type`

The active-loan write query uses alias `l` with `FOR UPDATE OF l`, matching the Rust lock semantics.

## Mapping Notes

`src/mappers.ts` maps database rows into domain entities and validates status values before casting them to domain status unions. Invalid database status values throw infrastructure errors.
